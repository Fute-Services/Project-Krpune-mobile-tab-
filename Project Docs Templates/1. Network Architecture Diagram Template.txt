1. PURPOSE
Document the network footprint of the KRPUNE Native mobile app, based on inspection of its source code.

2. SCOPE
Client-side mobile app only (Android/iOS). This app has no backend server of its own.

3. COMPONENTS FOUND IN CODE
- [x] End users / client devices (Android/iOS phones and tablets running the Expo app)
- [ ] Load balancers - none, no backend
- [ ] Web servers / application servers - none, app is fully offline/bundled
- [ ] Databases - none, all content is bundled as local JSON files (src/offline/data/*.json)
- [ ] Firewalls - not applicable, no server infrastructure to protect
- [ ] VPN / remote access points - none
- [x] Cloud services used - Cloudinary (CDN), used only to fetch static images/video assets over HTTPS
- [ ] On-premise servers - none
- [ ] Third-party integrations / APIs - none beyond the Cloudinary media CDN
- [ ] Internal network segments / VLANs - not applicable, single client app

4. NETWORK ZONES
There is no server-side infrastructure for this app, so there are no internal network zones to define. The only network boundary is: [Mobile App] --HTTPS--> [Cloudinary CDN] for media assets. All other data (floor plans, amenities, mobility, transport, gallery listing, VR tour metadata) ships bundled inside the app and requires no network call.

5. DIAGRAM ATTACHMENT
Given the above, the "diagram" is effectively a single arrow: Mobile App -> Cloudinary CDN (HTTPS, images/video only). A drawn version (box-and-arrow) can be produced on request, but there is no multi-tier infrastructure to depict.
